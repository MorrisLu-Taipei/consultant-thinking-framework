# 我們怎麼做出這本書:過程、流程、Lesson Learned

> 用途:教學「**如何用 AI 寫一本可操作化的方法論書**」。記錄這套系統的真實誕生過程、可複製流程、以及踩到的坑。
> 一句話:這本書不是一次寫成的,是**從一個被批評的想法,經過多輪『產出→被打臉→修正』長出來的**。

---

## 一、真實過程(它怎麼長出來的)

| 階段 | 發生什麼 | 關鍵動作 |
|---|---|---|
| 0 起點 | 一個「九宮格九維度」的想法 + 同事的尖銳校準(價值在對角線、要加我們自己、補沒說出口) | **先聽懂批評,不急著辯護** |
| 1 方法論 | 把想法寫成核心矩陣 + 三讀法 + C1-C5(`01`) | 把「概念」變「結構」 |
| 2 操作化 | 配上模板 `02`、評分 `03`、教練 `04`、範例 `08` | **產出≠理論,要能交件/評分/帶人** |
| 3 對標 | WebSearch 研究 3C/TOWS/Kano… 比較吸收(`05`) | 站在巨人肩上,不重造輪子 |
| 4 誠實校正 | **prior-art 查證**發現「不是首創」→ 改定位成「能力系統」(`06`) | **查過才敢說,不過度宣稱** |
| 5 實證設計 | 前後測 protocol `07` + 執行包 `15` | 從「概念」鋪路到「有證據」 |
| 6 自動化 | n8n 教練 Bot `16`(零阻力推廣) | 讓方法論被自動套用 |
| 7 主動書 | AI-native ebook `13` + `consulting-thinking-coach` skill | 不只被讀,可被用、可反問 |
| 8 互動化 | 單檔互動網站 `17`(免安裝、自評器、Webhook) | 把書變成可跑的工具 |
| 9 多語/防呆 | 7 語版 + 防酸民全文審查 | 對外發佈前的最後一哩 |

> 觀察:**每一個跳躍幾乎都來自一次外部批評**(同事/審稿視角)。這本書是被「打臉」打出來的。

---

## 二、可複製流程(你要寫一本這種書,照這個走)

```
① 一個核心洞見(可被一句話講完)
② 寫成結構(矩陣/階梯/流程)——讓它可被指認
③ 操作化三件套:模板(怎麼產出)+ 評分(怎麼判好壞)+ 教練(怎麼帶人)
④ 對標既有(WebSearch)——吸收,不重造
⑤ prior-art 查證——誠實定位,刪掉所有「首創/空白」
⑥ 鋪實證路徑(設計 + 執行包)——即使還沒跑
⑦ 做成「活的」:AI skill + 自動化 workflow
⑧ 做成「可跑的」:單檔互動網站(免安裝)
⑨ 多語 + 防酸民審查 + DOI
```
> 核心心法:**先求一個洞見的深度(②③),再求覆蓋與通路(④–⑨)。** 跟方法論本身一樣:先一條對角線,再九宮格。

## 三、用了哪些工具(AI-native 製書基礎設施)

- **AI IDE**(Claude Code / Codex / Antigravity):寫、改、跨檔一致性、多引擎對抗式審查。
- **WebSearch**(經本機 OpenAI 相容端點):對標研究 + prior-art 查證,**每個宣稱附來源**。
- **Skill 機制**:把方法論封裝成可被「講人話」觸發的教練。
- **單檔 HTML/JS**:把書變成免安裝、可離線、可跑的互動工具。
- **Git / 版本 / DOI**:可重現、可引用。
> 這就是 TigerAI「AI-native ebook」範式:書 = 由多 IDE 編排 + 版本控制 + 可執行互動共同構成的知識製品。

## 四、Lesson Learned(寫這種書的真教訓)

1. **先聽懂批評再動手。** 最強的升級(對角線、加我們自己、輸入vs產出、能力系統定位)全來自別人的校準,不是自己拍腦袋。
2. **prior-art 查證救了我們。** 一度寫「真空白地帶/首創」,查證後發現是 3C+TOWS+Temporal 的新組合 → 改誠實版本。**沒查就宣稱=給酸民把柄。**
3. **操作化 > 理論。** 純框架到處都是;真正稀有、最該主打的是「可訓練/可評分/可驗證的系統」。價值在 rubric+階梯+前後測,不在那張矩陣。
4. **做『活的』比寫『多的』有用。** 一個能跑的互動網站 + 一個 AI skill,勝過再加十份 PDF。降低使用阻力才會被用。
5. **誠實是賣點,不是弱點。** 主動寫限制、主動說「尚未實證」「不宣稱首創」,反而讓懂的人更信。
6. **區分輸入 vs 產出。** 一份報告值不值錢,看「產出格 + 對角線」,不是「輸入抄得整齊」。這個區分本身也是被審查者點出來的。
7. **多輪迭代留痕。** 每次被打臉就修一版,過程本身(本檔)就是教材。

## 五、給想教「怎麼寫這本書」的人
照 §二 流程帶學員做一輪,每階段對應本模組一個檔案當範例(①②→`01`、③→`02/03/04`、④⑤→`05/06`、⑥→`07/15`、⑦→`13`+skill、⑧→`17`)。
**最重要的教學點**:把「被批評」設計進流程——每寫完一版,**故意找一個酸民視角的人來挑**,挑出的洞就是下一版。這本書的品質,等於它被挑過幾輪。

## 六、後續延伸:從「一本書」長成「一個系列」(第二輪真實教訓)
這本書寫完後,我們發現它其實是**上集(顧問怎麼想)**,而既有的導入工具包是**下集(顧問案怎麼跑)**。於是又跑了一輪「系列化」,踩到新坑:

| 階段 | 發生什麼 | 關鍵動作 |
|---|---|---|
| 10 系列化 | 把上集 + 下集綁成「AI-Native 顧問方法論系列」,補橋接 `22`、系列總入口 `SERIES_INDEX`、路線圖 | 先建**敘事與入口**,不急著搬目錄(搬檔會斷連結/斷 DOI) |
| 11 名詞憲法 | 審查發現「內容長太快」→ 同一個 **L1-L5 被兩種意思共用**(上集顧問能力 vs 下集企業成熟度)、八階段被抄成目錄、Phase 定義在不同檔不一致 | 立 **`CANONICAL_TERMS_AND_MAPS.md` 憲法**,鎖死唯一定義,再回頭同步所有檔 |

### 新增 Lesson Learned(8–12)
8. **內容長太快會「撞名」。** 寫得多、寫得快,最容易讓同一個詞(L1-L5)在不同地方指不同東西。**規模一大,先立『名詞憲法』再繼續寫**,否則使用者被教錯。
9. **同一套符號不能有兩種意思——用 namespace 分流。** 顧問能力改 **C1-C5**、企業成熟度保留 **L1-L5**;一句話分流:「C 問人,L 問企業」。命名衝突要用前綴解決,不是靠上下文猜。
10. **互動頁/摘要最容易「抄成目錄」。** 把權威的「八階段(方法論本體)」抄成了「目錄章節順序」——看起來像,其實是兩張不同的圖。**列清楚四張圖各自用途**(服務流程/八階段/合約 Phase/接案生命週期),別混用。
11. **同一個概念散在多檔 → 必然漂移。** Phase A/B/C 在 `EIGHT_STAGE_FLOW_STORY` 與 `03_README` 寫得不一樣。**指定單一 canonical 來源,其他檔只引用、不各自定義。**
12. **授權別隨手宣稱。** 一度寫「文字 CC BY 4.0」但 repo 裡根本沒有那張授權檔——**宣稱即斷裂**。要嘛補檔定義清楚,要嘛全部維持單一授權。

> 第二輪總心法:**第一輪靠「被批評」長出深度;第二輪靠「立憲法」長出一致性。** 內容越多,殺手不是不夠好,是不一致。
